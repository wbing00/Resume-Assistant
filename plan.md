# Resume Assistant 开发计划

## 1. 开发目标

本计划面向 `Resume Assistant V1` 的实际落地开发，目标是在可控复杂度内完成一个可运行的闭环产品：
- 用户登录
- 上传简历
- 输入 JD
- 获得分析结果
- 生成定制化内容
- 记录投递结果
- 查看个人历史数据
- 管理员查看聚合数据

开发原则：
- 先打通闭环，再做体验优化
- 先保证数据结构和权限正确，再补功能细节
- 所有 AI 输出都尽量结构化
- V1 不做复杂 Agent，不做过度自动化

## 2. 总体开发顺序

推荐按以下顺序推进：
1. 环境准备
2. 项目初始化
3. 数据库与权限模型
4. 登录与基础页面框架
5. 简历上传与解析
6. JD 输入与解析
7. 匹配分析与内容生成
8. 投递记录与反馈回填
9. 管理看板与聚合指标
10. 埋点、测试与收尾
11. 部署与试用验证

## 3. 分阶段开发计划

## 阶段 1：环境准备

### 要做什么
- 注册 `Supabase` 账号
- 注册 `OpenAI API` 或兼容模型平台账号
- 准备本地开发环境
- 确认 Node.js、npm 可用
- 准备 `.env.local` 所需变量

### 产出
- 可访问的 Supabase 项目
- 可调用的模型 API Key
- 本地可运行的 Node.js 环境

### 验收标准
- 能登录 Supabase 控制台
- 能在本地读取环境变量
- 能成功发起一次模型 API 请求

## 阶段 2：项目初始化

### 要做什么
- 使用 `Next.js` 初始化项目
- 配置 `TypeScript`
- 配置 `Tailwind CSS`
- 引入 `shadcn/ui`
- 建立基础目录结构
- 初始化 Git 仓库或确认现有仓库结构

### 建议目录
- `app/`：页面与路由
- `components/`：UI 组件
- `lib/`：通用工具
- `lib/supabase/`：Supabase 客户端
- `lib/ai/`：模型调用与 prompt 封装
- `services/`：业务服务层
- `types/`：类型定义
- `supabase/`：SQL 和配置

### 产出
- 一个可以本地启动的前端工程
- 基础布局页和空白首页

### 验收标准
- `npm run dev` 能正常启动
- 页面可访问
- 基础样式和组件库正常工作

## 阶段 3：数据库与权限模型

### 要做什么
- 在 Supabase 中创建核心表
- 建立 `profiles` 表并加入 `role`
- 建立 `resumes`
- 建立 `job_descriptions`
- 建立 `analyses`
- 建立 `applications`
- 建立 `feedbacks`
- 建立 `events`
- 为各业务表配置 `user_id`
- 开启 `Row Level Security`
- 编写基础访问策略

### 关键点
- 普通用户只能访问自己的数据
- 管理员看板不直接暴露全表查询
- 角色字段默认 `user`
- 管理员账号手动改为 `admin`

### 产出
- 数据库表结构
- RLS 策略 SQL
- 初始种子数据方案

### 验收标准
- 用户登录后只能看到自己的数据
- 非管理员无法访问管理接口
- 数据表和字段满足产品文档要求

## 阶段 4：登录与基础框架

### 要做什么
- 接入 Supabase Auth
- 实现登录/退出
- 实现受保护页面
- 搭建整体布局
- 建立用户侧导航
- 建立管理侧导航
- 配置路由守卫

### 页面优先级
- `/login`
- `/dashboard`
- `/resume`
- `/analysis/new`
- `/applications`
- `/feedback`
- `/admin`

### 产出
- 完整的页面骨架
- 用户会话管理
- 基础权限跳转逻辑

### 验收标准
- 未登录用户不能进入业务页
- 非管理员不能进入 `/admin`
- 登录后能正常进入主工作台

## 阶段 5：简历上传与解析

### 要做什么
- 实现简历上传组件
- 上传文件到 Supabase Storage
- 保存文件元信息到 `resumes`
- 实现简历文本提取
- 将简历送入模型做结构化解析
- 保存 `parsed_text` 和 `structured_json`
- 展示简历解析结果
- 支持设置默认简历

### 技术要点
- V1 可先支持 PDF
- 文本提取可先选一个稳定方案
- AI 解析结果必须转为固定 JSON 格式

### 产出
- 用户可上传并管理简历
- 系统可展示结构化简历信息

### 验收标准
- 用户能成功上传文件
- 能看到解析后的结构化结果
- 默认简历状态正确保存

## 阶段 6：JD 输入与解析

### 要做什么
- 创建 JD 输入表单
- 支持手动粘贴 JD 文本
- 保存 JD 原文到 `job_descriptions`
- 调用模型解析岗位要求
- 输出结构化 JD 数据
- 页面展示岗位能力画像

### 解析重点
- 岗位名称
- 公司名称
- 核心职责
- 必备技能
- 加分项
- 经验要求
- 关键词

### 产出
- 用户可提交 JD
- 系统可输出结构化 JD 分析结果

### 验收标准
- JD 可正常保存
- 结构化字段完整度可接受
- 页面展示清晰可读

## 阶段 7：匹配分析与内容生成

### 要做什么
- 选择一份简历和一条 JD 发起分析
- 编写匹配分析 prompt
- 输出匹配得分、优势、差距、优化建议
- 生成定制版简历 bullet
- 生成自我介绍
- 生成投递沟通文案
- 保存结果到 `analyses`
- 做结果页展示

### 核心要求
- 不允许虚构用户经历
- 输出必须基于真实简历内容
- 给出“为什么这样修改”的解释
- 内容可复制、可编辑、可确认采纳

### 产出
- 一次完整的 AI 分析任务
- 一页可读、可操作的分析结果

### 验收标准
- 分析结果能落库
- 匹配解释具有可读性
- 生成内容可以直接用于投递准备

## 阶段 8：投递记录与反馈回填

### 要做什么
- 创建投递记录页
- 支持用户新增投递记录
- 关联分析记录
- 记录公司、岗位、渠道、投递时间
- 记录是否采用 AI 建议
- 更新投递状态
- 创建反馈表单
- 记录回复、面试、拒绝、Offer 等结果
- 保存主观评分和备注

### 闭环关键点
- 用户必须能把“AI 输出”连接到“真实投递结果”
- 投递结果和反馈要可追踪

### 产出
- 用户投递历史页
- 反馈记录能力

### 验收标准
- 投递信息可新增和更新
- 反馈提交后可回看
- 一次完整闭环数据链路打通

## 阶段 9：管理看板与聚合指标

### 要做什么
- 创建管理员专属页面 `/admin`
- 实现服务端角色校验
- 实现管理指标查询接口
- 聚合用户数、分析次数、投递数、反馈数
- 统计建议采纳率
- 构建基础漏斗
- 展示核心图表或指标卡片

### 指标建议
- 总注册用户数
- 完成简历上传的用户数
- 发起 JD 分析的用户数
- 创建投递记录的用户数
- 提交反馈的用户数
- 建议采纳率
- 回复率

### 产出
- 管理后台首页
- 基础业务指标面板

### 验收标准
- 只有 `admin` 可访问
- 指标数据与数据库一致
- 能支持 V1 阶段的产品验证

## 阶段 10：埋点、测试与收尾

### 要做什么
- 建立关键事件埋点写入 `events`
- 埋点用户关键行为
- 补充错误处理
- 补充 loading、empty、error 状态
- 清理无用代码
- 做基本的 UI 统一
- 补充必要提示文案
- 增加隐私说明和测试版提示

### 测试范围
- 登录流程
- 文件上传流程
- JD 解析流程
- 分析生成流程
- 投递记录流程
- 反馈回填流程
- 管理页访问权限

### 产出
- 可试用的稳定版本
- 关键埋点事件数据
- 基础体验可接受

### 验收标准
- 核心流程无阻塞错误
- 关键异常有提示
- 主要埋点可落库

## 阶段 11：部署与试用验证

### 要做什么
- 部署前端到 `Vercel` 或等价平台
- 配置线上环境变量
- 验证线上登录、数据库、存储、模型调用
- 邀请 5-10 个种子用户试用
- 收集反馈
- 记录用户的实际使用障碍
- 输出第一轮版本复盘

### 验证重点
- 用户是否能独立完成闭环
- 结果页是否真的有帮助
- 用户是否愿意回填投递反馈
- 哪一步最容易流失

### 产出
- 可访问的线上版本
- 第一轮试用反馈
- 后续优化清单

### 验收标准
- 线上版本可稳定访问
- 至少完成一轮真实用户验证
- 能形成下一版迭代方向

## 4. 开发中的注意事项

- 所有敏感 key 只放服务端，不暴露在前端
- 管理看板数据只允许管理员访问
- 简历与投递数据默认按用户隔离
- AI 输出必须允许用户手动确认
- 不要在 V1 过早引入复杂工作流或 Agent 编排
- 每做完一个阶段都做一次最小回归测试

## 5. 建议的执行方式

推荐按下面节奏推进：
1. 先完成阶段 1 到阶段 4，打好基础设施和权限
2. 再完成阶段 5 到阶段 7，跑通核心 AI 价值
3. 然后完成阶段 8 和阶段 9，补齐闭环和验证能力
4. 最后完成阶段 10 和阶段 11，做试用与迭代准备

## 6. 当前最优先事项

如果现在正式开始开发，最先做的是：
1. 注册并配置 `Supabase` 和模型 API
2. 初始化 `Next.js` 项目
3. 创建数据库表和权限策略
4. 先把登录和基础页面跑起来

只有这些完成后，后面的 AI 分析和闭环记录才有稳定落点。
