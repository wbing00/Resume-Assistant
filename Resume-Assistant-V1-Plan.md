# Resume Assistant V1 详细计划

## 1. 项目定位

### 1.1 项目名称
Resume Assistant V1

### 1.2 项目目标
面向正在求职的 AI 产品经理或互联网求职者，提供一个从 `JD 分析 -> 简历定制 -> 投递记录 -> 结果反馈` 的轻量闭环工具，帮助用户减少盲投，提高投递准备效率，并沉淀可复用的求职策略。

### 1.3 V1 核心原则
- 不追求复杂 Agent，不做过度自动化
- 先跑通用户核心任务闭环
- 先验证真实价值，再考虑复杂编排
- 技术选型优先服务于上线速度和结果验证

## 2. V1 要解决的问题

### 2.1 用户画像
- 目标用户：正在求职的 AI 产品经理、互联网产品经理、运营/策略等需要定制化投递的人群
- 使用场景：用户每天会浏览多个 JD，但很难快速判断匹配度，也缺乏针对岗位的简历改写和投递复盘机制

### 2.2 核心痛点
- 不知道一个 JD 真正看重什么能力
- 简历不知道该怎么针对岗位改写
- 每次投递都重复劳动，耗时高
- 投递后没有系统记录，无法复盘为什么没回复
- 现有简历工具偏向润色文本，缺少投递反馈闭环

### 2.3 V1 的价值假设
如果系统可以帮助用户快速拆解 JD、识别简历差距、生成定制化内容，并持续记录投递反馈，那么可以显著降低投递准备时间，并提升用户对投递策略的清晰度。

## 3. V1 成功标准

### 3.1 产品目标
- 让用户在一个产品内完成求职前期最关键的闭环动作
- 形成可追踪的用户行为和结果数据
- 为后续版本提供真实反馈样本

### 3.2 核心指标
- 单次 JD 分析和简历改写耗时下降
  - 目标：从 45-60 分钟下降到 10-20 分钟
- 用户完成一次完整流程的比例
  - 定义：上传简历 + 输入 JD + 生成结果 + 记录一次投递
- 建议采纳率
  - 定义：用户是否采用系统生成的简历/文案
- 投递反馈填写率
  - 定义：投递后是否回填结果
- 用户主观满意度
  - 定义：对输出结果是否有帮助的评分

### 3.3 V1 不承诺的结果
- 不承诺直接提升 offer 转化率
- 不承诺完全自动投递
- 不承诺替代用户做职业决策

## 4. 产品范围

### 4.1 V1 必做模块
- 用户登录
- 简历上传与管理
- JD 输入与解析
- 简历与 JD 匹配分析
- 定制化简历建议生成
- 投递记录管理
- 反馈回填
- 基础数据看板

### 4.2 V1 不做模块

- 浏览器插件抓取招聘网站
- 面试模拟语音功能
- 社交网络/人脉推荐
- 高级推荐算法

## 5. 用户核心闭环

### 5.1 主流程
1. 用户注册并登录
2. 上传一份基础简历
3. 粘贴目标 JD
4. 系统解析 JD 并提取岗位要求
5. 系统分析简历与岗位要求的匹配差距
6. 系统生成定制化简历建议和文案
7. 用户采纳内容并完成投递
8. 用户记录投递时间和结果
9. 系统根据反馈沉淀策略建议和历史数据

### 5.2 闭环定义
闭环不是“生成一次内容”，而是：
- 有输入：简历、JD、历史投递信息
- 有处理：AI 分析与生成
- 有行动：用户真的拿去投递
- 有反馈：用户记录回复情况
- 有沉淀：产品记住结果并服务下一次使用

## 7. 功能模块设计

### 7.1 简历解析模块
输入：
- 用户上传的简历文件

处理：
- 提取文本
- 识别教育、项目、实习、技能等结构
- 转为结构化 JSON

输出：
- 可展示的结构化简历数据
- 后续分析使用的标准化文本

### 7.2 JD 解析模块
输入：
- 用户粘贴的 JD 文本

处理：
- 提取岗位名称、职责、硬技能、软技能、经验要求、优先条件
- 给出岗位关注能力画像

输出：
- 结构化 JD 数据
- 岗位关键词
- 需求优先级

### 7.3 匹配分析模块
输入：
- 结构化简历
- 结构化 JD

处理：
- 做能力项比对
- 输出匹配度、缺口点、可以强化的经历

输出：
- 匹配得分
- 优势项
- 缺失项
- 优化建议

### 7.4 内容生成模块
输入：
- 匹配分析结果
- 用户原始简历内容

处理：
- 定向改写项目描述
- 生成针对岗位的简历 bullet
- 生成自我介绍文案
- 生成投递附言

输出：
- 多段可直接使用的文本内容

### 7.5 投递记录模块
输入：
- 用户手动填写投递信息

处理：
- 保存投递记录
- 绑定分析任务和简历版本

输出：
- 时间线化的投递历史

### 7.6 反馈回填模块
输入：
- 用户填写的投递结果
- 用户对建议的主观评分

处理：
- 更新投递状态
- 记录 AI 建议是否被采纳
- 记录实际结果

输出：
- 可用于复盘的数据样本

## 8. 技术栈规划

### 8.1 前端
- Next.js
- TypeScript
- Tailwind CSS
- shadcn/ui

选择原因：
- 上线快，工程组织清晰
- 页面型产品表达力更强
- 适合个人开发和后续扩展

### 8.2 后端与数据
- Supabase Auth
- Supabase Postgres
- Supabase Storage
- Supabase Row Level Security

选择原因：
- 登录、数据库、文件存储一体化
- 适合 MVP 快速搭建
- 降低基础设施成本

### 8.3 AI 能力层
- OpenAI API 或兼容接口模型
- 服务端封装统一 prompt 调用
- 所有输出尽量结构化 JSON

选择原因：
- 重点验证 AI 在流程中的价值，不做模型训练
- 便于后续替换不同模型

### 8.4 V1 是否使用工作流平台
- V1 第一阶段可不引入 n8n
- 先将核心逻辑写在 Next.js Server Actions 或 API Route 中
- 当需要做异步提醒、定时回访时，再引入 n8n

### 8.5 数据分析
- 第一阶段使用数据库聚合 + 自建 dashboard
- 不急于接入复杂埋点平台

## 11. Prompt 设计原则

### 11.1 Prompt 目标
- 保证结构化输出
- 降低幻觉
- 提高结果可解释性

### 11.2 关键策略
- 简历解析和 JD 解析都要求输出 JSON
- 匹配分析先比较，再打分，不直接空泛评价
- 文案生成要基于原始事实，不允许捏造经历
- 给用户展示“为什么这么改”的理由

### 11.3 基础校验
- 如果模型输出缺字段，做兜底重试
- 如果生成内容与原始经历不一致，提示用户手动确认

## 13. 权限模型与管理看板访问控制

### 15.1 权限设计目标
- 区分用户侧页面与管理侧页面
- 保护用户隐私，避免普通用户看到全局数据
- 确保管理看板仅供开发者或管理员查看
- 在页面层、接口层、数据库层都做权限校验

### 15.2 角色定义
- `user`：普通用户，只能访问自己的简历、JD、分析记录、投递记录和反馈数据
- `admin`：管理员，可以访问全局聚合数据、行为漏斗、建议采纳率、整体回复率等管理看板信息

### 15.3 页面权限范围
用户侧页面：
- Dashboard
- 简历管理页
- JD 分析页
- 匹配与改写结果页
- 投递记录页
- 反馈与复盘页

管理侧页面：
- `/admin`
- `/admin/dashboard`
- 其他仅用于查看全局运营数据的页面

约束原则：
- 普通用户不可见管理入口
- 即使知道管理路由，非管理员也无法访问

### 14.4 推荐实现方式
采用 `RBAC`，即基于角色的访问控制。

实现方案：
1. 在 `profiles` 表中维护用户角色字段
2. 新用户默认角色为 `user`
3. 开发者账号手动设置为 `admin`
4. 服务端在渲染管理页前先校验当前用户角色
5. 管理接口只对 `admin` 开放

### 14.5 数据表补充
建议新增一张 `profiles` 表，或在用户扩展信息表中增加如下字段：
- id
- email
- role
- created_at

角色字段说明：
- `role = user`
- `role = admin`

### 13.6 路由层访问控制
对于管理页面，必须在服务端做权限拦截，而不是只在前端隐藏按钮。

实现要求：
- 页面加载前读取当前登录用户
- 查询该用户的 `role`
- 如果不是 `admin`，则重定向到首页或返回 `403`

原因：
- 前端隐藏入口不等于安全
- 路由级校验可以避免普通用户直接访问管理地址

### 13.7 接口层访问控制
管理看板相关接口应与普通用户接口分离。

示例：
- 普通用户接口：`/api/applications`
- 管理接口：`/api/admin/metrics`
- 管理接口：`/api/admin/funnel`
- 管理接口：`/api/admin/events`

约束：
- 普通接口只返回当前用户自己的数据
- 管理接口返回全局聚合结果
- 非 `admin` 请求管理接口时直接拒绝

### 13.8 数据库层访问控制
使用 Supabase 的 `Row Level Security` 保护核心业务表。

适用表：
- `resumes`
- `job_descriptions`
- `analyses`
- `applications`
- `feedbacks`
- `events`

基础规则：
- 普通用户只能读取和写入 `user_id = auth.uid()` 的数据
- 管理员不通过用户侧表直接绕过规则读取明细数据，管理统计优先走服务端聚合

### 13.9 管理看板数据获取方式
V1 推荐采用服务端聚合方式，而不是直接在前端读取全表。

推荐链路：
- 管理员访问 `/admin`
- 服务端校验管理员身份
- 服务端使用安全权限查询聚合数据
- 返回给管理看板页面展示

这样做的原因：
- 结构更清晰
- 避免普通用户拿到全局数据查询能力
- 便于后续扩展更多管理指标

### 13.10 管理看板展示内容
管理看板建议展示聚合数据，而非单用户明细数据，重点包括：
- 总用户数
- 活跃用户数
- 简历上传次数
- JD 分析次数
- 建议采纳率
- 投递记录数
- 反馈填写率
- 使用漏斗
- 不同阶段转化率

### 13.11 面试表达重点
这部分在面试中可以这样表达：
- 我将用户侧和管理侧做了权限隔离
- 普通用户只能访问自己的求职数据
- 管理看板用于产品验证和运营分析，仅管理员可访问
- 我在页面层、接口层和数据库层做了三层访问控制，保证数据边界清晰

## 15. MVP 验证计划

### 15.1 种子用户
- 先找 5-10 个求职中的朋友试用
- 尽量覆盖不同岗位方向，但以产品经理为主

### 15.2 验证方式
- 每人至少完成 2 次 JD 分析和 1 次投递记录
- 收集主观满意度和是否采纳建议
- 观察是否真的减少准备时间

### 15.3 关键问题
- 输出结果是否足够具体
- 用户是否愿意真的用生成结果去投递
- 用户是否愿意回填反馈
- 闭环流程是否顺畅

## 16. 面试表达建议

这个项目在面试中不要只讲“我做了一个 AI 网页”，而要讲：
- 我围绕求职场景设计了一个从分析到反馈的闭环产品
- 第一版优先验证高频痛点和核心指标，而不是堆复杂 Agent
- 技术上选择了轻量可落地架构，确保能快速迭代
- 通过记录投递结果，把 AI 输出和真实结果连接起来，为后续策略优化提供依据

## 17. 下一步建议

V1 做完后，优先看这三件事：
- 用户是否持续使用
- 用户是否回填反馈
- 生成建议是否对结果有帮助

如果以上成立，再进入 V2：
- 加入自动提醒
- 加入定期复盘
- 加入更细的策略推荐
- 再考虑 Agent 化编排
